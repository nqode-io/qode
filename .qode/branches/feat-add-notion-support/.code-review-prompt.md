# Technical Specification — Add Notion Ticket Support

## 1. Feature Overview

Add a Notion ticket provider to qode so that teams using Notion databases as project boards can run `qode ticket fetch <notion-url>` to automatically extract page title and body content into `context/ticket.md`. This brings Notion to feature parity with the four existing providers (Jira, Azure DevOps, Linear, GitHub Issues), removing the manual copy-paste friction that currently blocks Notion-based teams from using the automated qode workflow.

**Business value:** Expands qode's addressable user base to teams using Notion for project management — a popular choice in startups and smaller organisations.

**Success criteria:**
- `qode ticket fetch` works with all common Notion URL formats
- Page content is converted to readable markdown
- Documentation and configuration references are updated
- All unit tests pass

## 2. Scope

### In scope
- New `NotionProvider` implementing the existing `Provider` interface
- URL parsing for `notion.so` and `notion.site` domains
- Notion API integration (page retrieval + block children)
- Block-to-markdown conversion for common block types
- One level of nested block fetching (toggle, callout children)
- Pagination for block children responses
- Authentication via `NOTION_API_KEY` environment variable
- Provider registration in the existing provider list
- Documentation updates (how-to guide, YAML reference, README)

### Out of scope
- Notion OAuth authentication (using internal integration tokens only)
- Fetching page comments or activity history
- Writing back to Notion (status updates, etc.)
- Listing all pages in a Notion database
- Rendering images, files, embeds, or other non-text block types
- Deeply nested blocks (beyond one level of recursion)

### Assumptions
- Users will create a Notion internal integration and share target pages with it
- Notion API version `2022-06-28` remains stable
- Ticket pages are typically small (< 100 blocks)
- The existing `Provider` interface is sufficient — no interface changes needed

### Data Flow

1. User runs `qode ticket fetch https://www.notion.so/workspace/My-Ticket-abc123def456...`
2. CLI loads `.env`, resolves project root, gets current branch
3. `DetectProvider()` iterates providers; `NotionProvider.CanHandle()` matches `notion.so` host
4. `NotionProvider.Fetch()` is called:
   a. `parseNotionURL()` extracts 32-hex-char page ID, formats as UUID
   b. `fetchPage()` calls `GET /v1/pages/{uuid}` with Bearer auth → extracts title from `Title`-type property
   c. `fetchBlocks()` calls `GET /v1/blocks/{uuid}/children` (paginated) → returns block array
   d. For blocks with `has_children: true`, fetches one level of child blocks
   e. `blocksToMarkdown()` converts blocks to markdown string
5. CLI writes `# {Title}\n\n{Body}\n` to `.qode/branches/{branch}/context/ticket.md`

#### Retrieve Page

```

  Authorization: Bearer {NOTION_API_KEY}
  Notion-Version: 2022-06-28

Response 200:
{
  "id": "uuid",
  "properties": {
    "Name": {            // property name varies; find by type == "title"
      "type": "title",
      "title": [
        { "plain_text": "My Ticket Title" }
      ]
    }
  }
}
```

#### Retrieve Block Children

```
GET https://api.notion.com/v1/blocks/{page_id}/children?page_size=100&start_cursor={cursor}
Headers:
  Authorization: Bearer {NOTION_API_KEY}
  Notion-Version: 2022-06-28

Response 200:
{
  "results": [
    {
      "type": "paragraph",
      "paragraph": {
        "rich_text": [
          {
            "plain_text": "Some text",
            "annotations": { "bold": false, "italic": false, "code": false, ... },
            "href": null
          }
        ]
      },
      "has_children": false
    }
  ],
  "has_more": false,
  "next_cursor": null
}
```

### Error Responses

| HTTP Status | Meaning | Error Message |
|---|---|---|
| 401 | Invalid or missing token | `"Notion API returned 401 — check NOTION_API_KEY\nSet it with: export NOTION_API_KEY=your-token"` |
| 403 | Integration lacks access | `"Notion API returned 403 — ensure the integration has access to this page\nShare the page with your integration at notion.so/my-integrations"` |
| 404 | Page not found or not shared | `"Notion page not found — ensure the page is shared with your integration\nManage integrations at notion.so/my-integrations"` |
| 429 | Rate limited | `"Notion API rate limited — try again in a few seconds"` |
| 5xx | Server error | `"Notion API returned {status}: {body}"` |

## 8. Security Considerations

### Authentication
- `NOTION_API_KEY` is read from `os.Getenv()` (populated via `.env` auto-loading)
- Token is sent only to `api.notion.com` (or the injected `apiBase` in tests)
- Token is never logged, printed, or included in error messages

### Input Validation
- URL host is validated against `notion.so` and `notion.site` before any API call (prevents SSRF)
- Page ID is validated as exactly 32 hex characters
- API response bodies are read with `io.LimitReader` (1 MB limit) to prevent memory exhaustion

### Data Sensitivity
- Notion page content may contain sensitive information — it is written only to the local `.qode/branches/{branch}/context/ticket.md` file, consistent with all other providers
- The `.env` file containing `NOTION_API_KEY` should be in `.gitignore` (documented in how-to guide)
