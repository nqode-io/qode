# Requirements Refinement — feat-support-apps-directory

## 1. Problem Understanding

### Restatement
Some monorepos organize their projects inside an `apps/` (or similar) subdirectory rather than placing them directly at the repository root. Currently, `qode init` only scans **one level deep** from the repo root when detecting topology and tech stacks. This means projects nested under `apps/frontend/`, `apps/backend/`, etc. are invisible to both `workspace.Detect()` and `detect.Composite()`. The feature must extend the scanning logic so that container directories like `apps/` are traversed to discover the actual project directories within them.

### User Need & Business Value
Developers working in monorepos with an `apps/` (or `packages/`, `services/`, `projects/`) directory structure cannot use `qode init` out of the box — the tool detects the wrong topology (typically `single` instead of `monorepo`) and misses all tech layers nested inside the container directory. This forces manual editing of `qode.yaml`, which defeats the purpose of automatic detection. Fixing this makes `qode init` work correctly for a common monorepo pattern (Turborepo, Nx, Lerna, Rush, custom setups).

### Ambiguities & Open Questions (Resolved)
1. **Which container directory names should be recognized?** Per the notes, all well-known names across monorepo toolchains: `apps`, `packages`, `libs`, `libraries`, `services`, `projects`.
2. **Should the depth be configurable?** No. Fixed at one extra level (root → container → project). This covers all standard monorepo tools.
3. **Should `qode init --new --scaffold` also support creating the `apps/` structure?** Out of scope — the ticket only mentions `qode init` for existing projects.
4. **What about workspace config files (`turbo.json`, `nx.json`, `pnpm-workspace.yaml`)?** Out of scope for this ticket. However, the presence of these files at the root is a strong signal that the repo is a monorepo. We will use their presence as an **additional monorepo signal** in topology detection (simple file-exists check, no parsing required), without relying on them for container directory discovery.

### Key Technical Decisions

#### Decision 1: Where to define the shared container directory list

Both `workspace.Detect()` and `detect.Composite()` need the same set of known container directory names. Since `workspace` and `detect` are sibling packages with no import relationship, the options are:

- **Option A**: Define in a new shared package (e.g., `internal/fsutil/containers.go`) — clean but creates a new package for a single constant.
- **Option B**: Define in `detect` package (which is lower-level) and import from `workspace` — creates a dependency.
- **Option C**: Duplicate the list in both packages — simple, easy to test independently, 6 strings.

**Chosen: Option C** — duplicate the list. The list is small (6 entries), stable, and unlikely to change. Adding a shared package or cross-package import for a single `[]string` constant is over-engineering. Both packages already have no dependency on each other and should remain independent. If the list grows significantly in the future, it can be extracted then.

Each package defines its own:
```go
var knownContainerDirs = map[string]bool{
    "apps": true, "packages": true, "libs": true,
    "libraries": true, "services": true, "projects": true,
}
```

#### Decision 2: Heuristic for `apps/package.json` (container with tech markers)

A container directory like `apps/` may itself contain a `package.json` when it serves as a **workspace root** (e.g., Turborepo puts a root `package.json` with a `"workspaces"` field). The heuristic:

1. If a known container dir has tech markers (`looksLikeProjectDir` returns true) **AND** also has subdirectories that look like projects → treat it as a **container** (recurse into children). The container-level tech markers are a workspace root artifact, not a real project.
2. If a known container dir has tech markers but **no** project-like children → treat it as a **project** (do not recurse). It happens to be named `apps` but is actually a standalone project.
3. If a known container dir has no tech markers → treat it as a **container** (recurse into children).

Implementation in pseudocode:
```go
if knownContainerDirs[e.Name()] {
    children := scanChildren(subAbs)
    if len(children) > 0 {
        // Has project-like children → it's a container
        techDirCount += len(children)
    } else if looksLikeProjectDir(subAbs) {
        // No project children but has tech markers → it's a project
        techDirCount++
    }
    // else: empty container → skip
}
```

#### Decision 3: Monorepo signal files

The presence of `turbo.json`, `nx.json`, or `pnpm-workspace.yaml` at the repo root is a strong indicator of a monorepo. Rather than parsing these files, we add a simple file-exists check in `workspace.Detect()`:

```go
monorepoSignals := []string{"turbo.json", "nx.json", "pnpm-workspace.yaml", "lerna.json"}
for _, signal := range monorepoSignals {
    if fileExists(root, signal) {
        hasMonorepoSignal = true
        break
    }
}
```

This is used as a **tiebreaker**: if `techDirCount == 1` but a monorepo signal file exists, we still classify as `monorepo` rather than `single`. This handles the edge case where only one project lives inside `apps/` (a monorepo with one app currently).

#### Decision 4: Path representation and layer naming

- Paths: `./apps/frontend` (two-segment relative path from root). The `detectAt()` function already accepts arbitrary `relPath` strings, so `detectAt(subAbs, "./"+containerName+"/"+childName, 2)` works.
- Layer names: The existing `suggestName()` function uses `filepath.Base(path)` for depth > 0, which yields `frontend` from `./apps/frontend`. This is correct — layers should be named after the leaf directory, not the container. For depth 2 entries, we pass `depth=2` to `suggestName()` but the logic is the same (uses `filepath.Base`).

### Patterns/Conventions to Follow

- The existing skip-list in `detect.Composite()` (`node_modules`, `vendor`, `dist`, `build`, `.git`, `__pycache__`, `bin`, `obj`) must also apply when iterating children of container dirs.
- The existing `looksLikeProjectDir()` check is reused for children of container dirs.
- Confidence scoring and stack superseding logic in `detect.go` work unchanged since `detectAt()` already accepts arbitrary paths.
- Hidden directory filtering (`strings.HasPrefix(e.Name(), ".")`) applies inside container dirs too.

## 3. Risk & Edge Cases

### What Could Go Wrong
- **False positive container dirs**: A directory named `apps` that is itself a standalone project (not a workspace root). **Mitigation**: Apply the heuristic from Decision 2 — only recurse if container has project-like children.
- **Performance**: Adding a second level of scanning for ~6 known names. **Mitigation**: Only scan children of recognized container names. Each additional `os.ReadDir` call is O(entries) and negligible for typical monorepos (<50 entries per container).
- **Layer name collision**: Two containers (`apps/web` and `packages/web`) would both produce a layer named `web`. **Mitigation**: The dedup logic in `dedup()` handles same-stack-same-name collisions. For different stacks at different paths, both are kept — the names differ by path. If both are the same stack, the higher-confidence one wins. This is acceptable; users can rename layers in `qode.yaml`.

### Edge Cases
1. **`apps/` exists but is empty or has no project children** → Ignored. `looksLikeProjectDir(apps/)` is false and no children found → `apps/` does not contribute to `techDirCount`.
2. **`apps/` itself has a `package.json` with `"workspaces"` field** → Container heuristic applies: if children are project-like, recurse. The workspace root `package.json` is not treated as a standalone project.
3. **`apps/` itself has a `package.json` but no project children** → Treated as a regular project (`techDirCount++`). It's a standalone project that happens to be named `apps`.
4. **Nested containers**: `apps/group/subapp/` — only one extra level. `group/` inside `apps/` is not itself in `knownContainerDirs` (unless it's named `packages` etc.), so `subapp/` is not discovered. This is by design — arbitrary nesting is out of scope.
5. **Symlinks inside `apps/`** → `os.ReadDir` returns `DirEntry` that does not follow symlinks. Consistent with existing behavior.
6. **Mixed layout**: Root-level projects AND container-nested projects → Both are detected. Deduplication handles overlapping stacks.
7. **Hidden directories inside container** (e.g., `apps/.internal/`) → Skipped by `strings.HasPrefix(e.Name(), ".")`.
8. **Container dir in skip-list**: If someone names a directory `build` and it's also a known container dir → the skip-list takes precedence (already excluded before container check). No conflict since `build` is not in `knownContainerDirs`.
9. **Only one project inside `apps/`** → Without monorepo signal files, `techDirCount == 1` → `TopologySingle`. With `turbo.json` present, the signal-file tiebreaker upgrades it to `TopologyMonorepo`.

### Security Considerations
- None. This is local filesystem scanning with no user input beyond the working directory.

### Performance Implications
- Minimal. At most 6 additional `os.ReadDir` calls (one per known container name if all exist). Total added I/O is negligible.

## 4. Completeness Check

### Acceptance Criteria
1. `qode init` on a repo with structure `apps/<project1>/ + apps/<project2>/` correctly detects topology as `monorepo`.
2. `qode init` discovers all tech layers inside known container dirs (`apps`, `packages`, `libs`, `libraries`, `services`, `projects`).
3. Generated `qode.yaml` contains layers with correct paths like `./apps/frontend`, `./packages/shared`.
4. Layer names use the leaf directory name (e.g., `frontend`, not `apps`).
5. Existing single-project and flat-monorepo detection is not regressed.
6. Container dirs that are themselves projects (no nested children) are still detected as projects.
7. Monorepo signal files (`turbo.json`, `nx.json`, `pnpm-workspace.yaml`, `lerna.json`) act as tiebreakers for topology detection.

### Implicit Requirements
- Skip-list for excluded directories applies inside container dirs.
- Hidden directory filtering applies inside container dirs.
- Layer deduplication works correctly for layers at mixed depths (root, depth-1, depth-2).
- The `--new` and `--workspace` init modes are not affected.

### Out of Scope
- Parsing workspace config files for glob patterns (no YAML/JSON parsing of `turbo.json` etc.).
- Scaffolding `apps/` structure via `qode init --new --scaffold`.
- Supporting arbitrary nesting depth beyond root → container → project.
- Multi-repo workspace changes (`runInitWorkspace`).
- Making the container directory list configurable via `qode.yaml`.
