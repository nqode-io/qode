# Requirements Refinement — bug-ticket-fetch-does-not-apply-dot-env

## 1. Problem Understanding

**Restated Problem:**
When a user runs the `/qode-ticket-fetch` slash command from within a VS Code or other IDE extension, the underlying `qode ticket fetch <url>` CLI command cannot access secrets like `GITHUB_TOKEN`. This is because the CLI reads tokens via `os.Getenv()` but never loads a `.env` file — so tokens only exist if the user explicitly exports them in a shell session. IDE-launched processes typically do not inherit the user's interactive shell environment.

**User Need & Business Value:**
Developers should be able to store API tokens in a `.env` file at the project root and have all `qode` CLI commands transparently pick them up. This eliminates the need for manual shell exports and makes the tool usable from IDEs without extra setup.

**Ambiguities:**
- Should `.env` loading be applied globally at program startup, or per-command?
- Should `.env` override existing env vars, or only fill in missing ones? (convention: `.env` should NOT override existing system/shell env vars)
- Should we search for `.env` in the project root (resolved via `qode.yaml` location), the working directory, or both?
- Are there other CLI subcommands that also use `os.Getenv()` directly that need the same fix? (Quick scan: only ticket providers do today, but the ticket says "apply to all slash commands")

## 2. Technical Analysis

**Affected Components:**

| Component | Path | Role |
|-----------|------|------|
| Main entry point | `cmd/qode/main.go` | Best single place to load `.env` early |
| Ticket fetch command | `internal/cli/ticket.go:23-69` | Calls provider which uses `os.Getenv()` |
| GitHub provider | `internal/ticket/github.go:93` | `os.Getenv("GITHUB_TOKEN")` |
| Jira provider | `internal/ticket/jira.go:43-44` | `os.Getenv("JIRA_API_TOKEN")`, `os.Getenv("JIRA_EMAIL")` |
| Linear provider | `internal/ticket/linear.go:28` | `os.Getenv("LINEAR_API_KEY")` |
| Azure DevOps provider | `internal/ticket/azuredevops.go` | `os.Getenv("AZURE_DEVOPS_PAT")` |
| Config loader | `internal/config/config.go:20-45` | Resolves project root — useful for locating `.env` |
| Claude dispatcher | `internal/dispatch/claude.go:49-82` | Passes `os.Environ()` to subprocess, so loading `.env` early fixes this too |

**Key Technical Decisions:**

1. **Where to load `.env`:** Load it in `main()` before `cli.Execute()` (in `cmd/qode/main.go`). This ensures all subcommands and all providers see the populated env vars without modifying every provider individually.

2. **`.env` file location resolution:** The project root is determined by walking up from `cwd` until a `qode.yaml` is found (see `internal/config/config.go`). Load `.env` from that same directory. If no project root is found (e.g., `qode init`), fall back to `cwd`.

3. **Non-override semantics:** Standard `.env` convention: only set an env var if it is not already set in the environment. This preserves CI/shell-exported values.

4. **Library choice:** Use `github.com/joho/godotenv` — the canonical Go `.env` library. It is not currently a dependency, so it must be added. Alternatively, implement a minimal parser inline to avoid a new dependency. Given the well-defined format of `.env` files, using `godotenv` is preferred for correctness.

5. **Silent failure when `.env` missing:** If no `.env` file exists, proceed silently. This is not an error.

**Patterns to Follow:**
- Config loading is done in the `internal/config` package. `.env` loading should live nearby or in a new `internal/env` package, keeping `main.go` thin.
- Error handling is explicit throughout; `.env` parse errors should log a warning but not fatal-exit.

**Dependencies:**
- New external dependency: `github.com/joho/godotenv` (or inline parser)
- No changes to ticket providers themselves — they continue using `os.Getenv()`

## 3. Risk & Edge Cases

| Risk | Mitigation |
|------|-----------|
| `.env` overrides CI secrets set by CI system | Use non-override load (`godotenv.Overload` vs `godotenv.Load`; use `Load` which does not override) |
| `.env` contains a token with a trailing newline or quote | `godotenv` handles this correctly |
| User has `.env` in home dir but not project root | Only search project root and cwd; do not walk upward past project root |
| `GITHUB_TOKEN` in `.env` is a real secret accidentally committed | Already a user responsibility; document `.env` should be in `.gitignore`. Check if `.gitignore` already includes `.env` |
| Multiple `.env` files (`.env.local`, `.env.production`) | Out of scope for this fix; only load `.env` |
| `qode init` run outside any project | No `qode.yaml` exists; fall back to cwd for `.env` search — safe |
| `.env` file is world-readable | Warn if file permissions are too open (optional/nice-to-have, not required) |
| Circular env var reference (e.g., `FOO=$BAR`) | `godotenv` handles basic variable expansion; edge case, not a blocker |

**Security Considerations:**
- `.env` files should not be committed. Verify `.gitignore` includes `.env` (currently the project has a `.env` file — check it is in `.gitignore`).
- Tokens loaded from `.env` are placed into the process environment; they are not persisted anywhere else.
- The Claude dispatcher in `internal/dispatch/claude.go` already passes full `os.Environ()` to the subprocess, which is correct and means tokens become available to AI invocations as well.

**Performance:** Negligible. File read of a small `.env` file at startup has no meaningful cost.
