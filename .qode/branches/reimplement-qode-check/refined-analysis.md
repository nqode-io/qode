# Refined Analysis: Replace `qode check` CLI with `/qode-check` IDE Slash Command

## 1. Problem Understanding

The current `qode check` CLI command runs quality gates (tests → lint → AI review scores) as a dumb sequential script: it executes shell commands from `qode.yaml`, reads previously-saved AI review score files, and exits with pass/fail. It cannot reason about failures, propose fixes, or interact with the developer.

The goal is to replace this with `/qode-check` — an AI-driven interactive quality gate slash command that runs inside the IDE (Claude Code and Cursor). The AI discovers and runs all unit tests and the linter from the project structure, interprets failures, and offers the developer Accept/Stop/Comment options at each failure point.

**Workflow position:** `/qode-start → (implementation) → /qode-check → /qode-review-code → /qode-review-security`

**User need:** A quality gate step between implementation and code review that uses AI to run tests/lint, understand failures, and propose fixes — without context-switching to a terminal.

**Business value:** Catches and fixes regressions and lint violations before code review, reducing review cycle time.

**Resolved decisions (from notes.md):**
- The `/qode-check` prompt lives directly in the slash command files generated by `claudecode.go` and `cursor.go` — no `internal/prompt/templates/check/base.md.tmpl`, no `.qode/prompts/` override path.
- Phase 1: AI runs all unit tests. It determines how by inspecting the project structure — **does not read `qode.yaml`**.
- Phase 2: AI runs the linter. It determines how by inspecting the project structure — **does not read `qode.yaml`**.
- `qode.yaml` is consumed exclusively by the `qode` CLI tool. No AI-facing prompt or slash command may instruct the AI to read it (fields like `test.unit`, `test.lint` are off-limits).
- `config.ReviewConfig` (`MinCodeScore`, `MinSecurityScore`) is kept — will be used by issue #30.

**Open questions:** None. All design decisions are resolved.

### Key technical decisions

**1. Slash command content is fully inline in `claudecode.go` / `cursor.go`**

The `qode-check` entry in `claudeSlashCommands()` and `slashCommands()` contains the complete Phase 1/2 behavior instructions as a Go string literal — identical in approach to the `"qode-knowledge-add-context"` entry (`claudecode.go:97–122`, `cursor.go:178–203`). No prompt engine, no template file, no local override path.

**2. Phase 1: AI detects and runs all tests from project structure**

The prompt instructs the AI to inspect the project for known test runner signals and run all tests:
- `go.mod` present → `go test ./...`
- `package.json` with `test` script → `npm test` (or `yarn test`)
- `pytest.ini` / `pyproject.toml` with pytest config → `pytest`
- Other stacks: AI applies equivalent discovery

No reference to `qode.yaml` or `test.unit`.

**3. Phase 2: AI detects and runs the linter from project structure**

The prompt instructs the AI to inspect the project for known linter signals:
- Go project (`go.mod` + `.golangci.yml` or `golangci-lint` in PATH) → `golangci-lint run`
- Node/TS project (`eslint` in `devDependencies` or `.eslintrc*`) → `npx eslint .`
- Python (`ruff.toml` / `pyproject.toml` with ruff) → `ruff check .`
- Other stacks: AI applies equivalent discovery

No reference to `qode.yaml` or `test.lint`.

**4. Runner package deletion is safe**

`internal/runner/runner.go` is imported only by `internal/cli/check.go` (confirmed: no other Go file imports `github.com/nqode/qode/internal/runner`). The `scoring` package used by `runner.go` is also used by `internal/plan/refine.go` and is kept.

**5. `config.ReviewConfig` is kept unchanged**

`MinCodeScore` and `MinSecurityScore` remain in `internal/config/schema.go:77-78` and `internal/config/defaults.go:10-11`. Kept for issue #30.

### Patterns to follow

- Inline command content: `"qode-knowledge-add-context"` in `claudecode.go` and `cursor.go` is the direct analogue.
- Claude Code `.md` format: plain markdown, no frontmatter.
- Cursor `.mdc` format: YAML frontmatter `---\ndescription: ...\n---\n` + markdown.
- Cursor workflow rule: `workflowRule()` `cursor.go:69` — `7. Check → qode check` → `7. Check → /qode-check`.

### Dependencies

No new packages or external dependencies. Prompt engine explicitly not used.

## 3. Risk & Edge Cases

**Risk 1 — Generated files must be regenerated and committed**
`.claude/commands/*.md` and `.cursor/commands/*.mdc` are checked into the repo. The PR must include the output of `qode ide setup`; otherwise the slash command won't exist for users who pull without running setup.

**Risk 2 — Root `Long` help text references `qode check`**
`internal/cli/root.go:44` — must update `9. qode check` to `/qode-check (in IDE)`.

**Risk 3 — Cursor workflow rule stale until regenerated**
`cursor.go:workflowRule():69` emits `7. Check → qode check` into `.cursorrules/qode-workflow.mdc`. Regenerated file must be in the PR.

**Risk 4 — AI unable to detect test runner or linter**
If no known signal is found (unfamiliar stack, non-standard layout), the AI must report it cannot determine the runner rather than guessing. The prompt must instruct: on unknown stack, report what was inspected and ask the user to specify the command.

**Risk 5 — Multi-layer projects**
Phase 1 must run tests for each layer and collect all failures before presenting the summary. Same for Phase 2. The prompt must make this explicit.

**Risk 6 — Accept loop must re-run the same phase**
Accept applies fixes and re-runs the failing phase — does not advance. The prompt must make this explicit.

**Edge cases:**
- No test files found → skip Phase 1, proceed to Phase 2, note the skip
- No linter found → skip Phase 2, report success
- All tests pass, lint clean → confirm success, suggest `/qode-review-code`
- Tests pass, lint fails → Phase 2 summary only; no Phase 1 re-run
- User chooses Stop → AI exits without changes
- User chooses Comment → AI incorporates comment before retry; no blind loop
- Multiple failures across layers → summarise all before presenting Accept/Stop/Comment

**Security:** No concerns — slash command file contains only AI instructions.

**Performance:** Removing `runner.go` reduces binary size marginally. No runtime implications.
