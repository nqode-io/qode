# Notes

## Decisions

**`/qode-check` prompt is inline — no template file, no `.qode/prompts/` override**
The Phase 1/2 behavior instructions live directly in the slash command files (`.claude/commands/qode-check.md` and `.cursor/commands/qode-check.mdc`), generated by `claudecode.go` and `cursor.go`. There is no `internal/prompt/templates/check/base.md.tmpl` and no support for local override via `.qode/prompts/check/base.md.tmpl`. Users who want different behavior can edit the generated command files directly.

**`/qode-check` Phase 1 runs all tests, not just affected paths**
Phase 1 runs all unit tests that are present in the project, not a subset based on execution path analysis. Do not use the `test.unit` command.

**Keep `config.ReviewConfig` (`MinCodeScore`, `MinSecurityScore`)**
These fields are not removed when `qode check` is deleted. They will be used by issue #30 (enforcing qode workflow steps strictly).

**AI must not read `qode.yaml` in prompts or slash commands**
`qode.yaml` is consumed exclusively by the `qode` CLI tool. AI-facing prompts and slash commands must not instruct the AI to read or parse it. The config will be simplified in issue #29 — test/lint command configuration will be removed and left to the developer to define via their own `CLAUDE.md`. New prompts and commands must not reference `qode.yaml` fields such as `test.unit`, `test.lint`, etc. The AI must determine how to run tests and lint from the project structure itself.

**Documentation must be updated alongside code changes**
When removing or replacing CLI commands, update all documentation in the same PR: README.md (workflow steps + commands section + slash command list), CONTRIBUTING.md (workflow snippet + quality gates section + submitting changes steps), docs/qode-yaml-reference.md (field descriptions referencing removed commands), and ROADMAP.md (move completed issues to Done, add in-progress items). Generated files (.cursorrules/qode-workflow.mdc) are updated by re-running `qode ide setup`.
