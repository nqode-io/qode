# Requirements Refinement — remove-prompt-only-flag

## 1. Problem Understanding

**Restated problem:** qode currently has two execution modes for commands like `qode plan refine`, `qode plan spec`, `qode start`, and `qode review *`: (a) interactive mode, which launches a Claude Code CLI session and runs the generated prompt inside it; and (b) `--prompt-only` mode, which only generates the prompt file and prints its path. The IDE slash commands (e.g. `/qode-plan-refine`) rely on `--prompt-only` to generate the prompt file, then read and execute it directly — bypassing the interactive Claude CLI invocation entirely. Since the interactive mode (launching Claude CLI) is redundant — IDEs already provide the LLM context — it should be removed. The ticket goes one step further and proposes outputting the prompt to `os.Stdout` instead of a file, letting the LLM read it inline.

**User need:** Developers using any IDE with any LLM should be able to run qode slash commands without qode needing to know about the AI runtime. qode's responsibility ends at generating a well-formed prompt; the IDE/LLM handles execution.

**Business value:** Removes IDE lock-in, simplifies the qode binary (no dispatch dependency at runtime), and makes the workflow clearer and more portable.

**Chosen approach — stdout output + --to-file flag:**
The ticket presents two options. The stdout approach is explicitly labelled "even better" and is feasible because the slash commands are executed by an LLM that can run a terminal command and read its stdout output. This is the recommended approach and drives all decisions below.

**Ambiguities / open questions:**
- `knowledge add-branch` has `--prompt-only` but is listed among commands that "should not be changed" in Additional Context. However it IS in `knowledge_cmd.go` which is not `knowledge` or `config` — need to confirm: does "knowledge and sub-commands should not be changed" cover `knowledge add-branch`? The ticket lists `qode knowledge` under "should not be changed", so `knowledge add-branch` should be preserved as-is.
- `qode review all` is to be removed — confirm it is not referenced anywhere externally (docs, CI, user scripts) before deletion.
- The TODO comment about a future `--force` flag for ide setup/sync overwrite behavior needs a precise location to be useful.

### Key technical decisions

1. **stdout as primary output:** `qode plan refine` (and all affected commands) will write the generated prompt to `os.Stdout` when called without flags. The slash command invokes the command, reads stdout, and uses it as the prompt.

2. **`--to-file` flag:** When passed, the command applies the template and saves the result to the existing prompt file path (`.qode/branches/<branch>/.<type>-prompt.md`). Errors during template execution are written to `os.Stderr` and the process exits non-zero. This replaces the `--prompt-only` flag entirely.

3. **Remove `qode review all`:** It is no longer needed — the slash commands for code and security review can be run independently. Before removing, check `README.md`, `docs/`, and any CI configs for references.

4. **Remove `internal/dispatch/` package:** Once no command calls `dispatch.RunInteractive()` or `dispatch.Run()`, the entire package becomes dead code. Confirm with a grep, then delete `dispatch.go`, `claude.go`, `claude_test.go`. Update any import in `internal/cli/` files.

5. **IDE slash command update:** The generated slash command bodies currently contain text like:
   ```
   First, run: qode plan refine --prompt-only
   Then read the file at: .qode/branches/$BRANCH/.refine-prompt.md
   ```
   These should be replaced with:
   ```
   Run: qode plan refine
   Use the stdout output as the prompt.
   ```
   This change lives in `internal/ide/claudecode.go` and `internal/ide/cursor.go` — in the embedded string templates for each slash command.

6. **`qode ide setup` and `qode ide sync` overwrite behavior:** Both should unconditionally overwrite existing slash command files. Add a `// TODO: implement --force flag before beta; currently always overwrites` comment at the point where the write decision is made (likely in `internal/ide/ide.go` or the individual IDE setup functions).

7. **`knowledge add-branch`:** The ticket lists `qode knowledge` under "should not be changed" — preserve `--prompt-only` on `knowledge add-branch` unchanged.

### Patterns and conventions to follow

- All errors returned via `fmt.Errorf("context: %w", err)` — maintain this.
- Flag definitions use `cmd.Flags().BoolVar(&flag, "name", default, "desc")` — use same for `--to-file`.
- Functions max 50 lines; if the updated run function exceeds this, extract helpers.
- No TODOs in committed code except the specific one requested for `--force` flag.

### Dependencies

- No external service dependencies.
- `internal/prompt` package (template engine) is not affected — it generates the prompt string; only the output destination changes.
- `internal/runner` package (quality gates for `qode check`) is not affected.

## 3. Risk & Edge Cases

| Risk | Mitigation |
|---|---|
| Slash commands in user repos still reference `--prompt-only` | `qode ide sync` will overwrite them; document this in release notes |
| `qode review all` removal breaks user workflows | Check README and docs first; note removal in changelog |
| stdout output mixed with log/debug output | Ensure all non-prompt output (progress messages, errors) goes to `os.Stderr` not `os.Stdout` |
| `--to-file` generates a partial file on template error | Write to a temp file, then rename on success; discard on error |
| dispatch package deletion breaks build | Confirm no remaining imports with `go build ./...` after removal |
| Lessons learned files reference old behavior | `wire-stdin-stdout-for-interactive-cli-dispatch.md` is almost certainly obsolete; review and delete or update |

**Security:** No security implications — removing a flag and changing output destination does not introduce attack surface.

**Performance:** Slight improvement — no subprocess launch for the common case.

### Implicit requirements

- All non-prompt output from affected commands must go to `os.Stderr` to keep stdout clean for LLM consumption.
- The `--to-file` flag should print the saved file path to `os.Stderr` (not stdout) for human feedback.
- The workflow documentation (`qode workflow` command output and docs) should clearly state that `qode plan refine`, `qode plan spec`, `qode start`, and `qode review *` are intended to be called from an IDE via slash commands, not directly from the terminal.

### Out of scope

- Implementing the `--force` flag for `ide setup`/`ide sync` (only the TODO comment is added).
- Changes to `qode init`, `qode branch`, `qode ticket fetch`, `qode plan status`, `qode check`, `qode knowledge`, `qode config`.
- Supporting other LLM runtimes beyond stdout-based invocation.
