## Ticket / Feature Description


# FEAT: Remove --prompt-only flag and make that the default behavior

## Problem / motivation

qode is meant to be used as a support tool in the process of writing code. While some functionalities should be called directly by the user from the terminal, most should only ever be used by AI.

In the beginning, commands like `qode plan refine` and `qode plan spec` were used to generate a prompt file, then drop into Claude Code CLI and execute the prompt. The slash command equivalents, namely `/qode-plan-refine` and `/qode-plan-spec`, instead first called `qode plan refine --prompt-only` (or `qode plan spec --prompt-only`) to generate the corresponding prompt file then executed the prompt directly.

Entering the Claude Code CLI is superfluous as the engineer's choice of IDE should be irrelevant and we also want to be able to support all IDEs and all LLMs.

## Proposed solution

We remove the call to the Claude Code CLI in qode altogether. Instead we are just left with the generated prompt file. As this is equivalent to always calling the command with `--prompt-only` flag this flag should also be removed.

The solution must then:

- Change the behavior of the command called (`qode plan refine`, `qode plan spec`, `qode start`, `qode review code`, `qode review security`, `qode review all` -- last of which should be removed as it is no longer necessary). All other commands that accept `--prompt-only` should be reviewed and a decision per case should be made as to how to deal with them
- Update the slash commands to reflect the removal of the `--prompt-only` flag
- Update all documentation which is applicable which should remove all uses of `--prompt-only` flag but also any mention of the use of affected terminal commands as the only proper way to use the tool from now on is through an IDE

## Alternatives considered

While keeping the current behavior where the prompt is generated and saved to its own prompt file and then executed by the slash command is acceptable, an even better solution would be to output the prompt directly to the `os.Stdout`. This is only applicable if the LLM can be instructed to call a terminal command, read its output and use it as a prompt. In this case adding a new `--to-file` flag would be useful for debugging the prompt templates if changed.

The solution must then:

- Change the behavior of the command called (`qode plan refine`, `qode plan spec`, `qode start`, `qode review code`, `qode review security`, `qode review all` -- the last of which should be removed as it is no longer necessary) to write the generated prompt to `os.Stdout` instead of to a prompt file. All other commands that accept `--prompt-only` should be reviewed and a decision per case should be made as to how to deal with them
- For the same commands add a `--to-file` flag to generate the prompt by applying the data to the template then save it to the prompt file as it does currently and if there are any errors while applying the data to the template any errors should be caught and written out to `os.Stderr` before gracefully terminating the process.
- Update the slash commands to reflect the removal of the `--prompt-only` flag
- Update all documentation which is applicable which should remove all uses of `--prompt-only` flag but also any mention of the use of affected terminal commands as the only proper way to use the tool from now on is through an IDE, and addition of the `--to-file` flag.

# Requirements Refinement — remove-prompt-only-flag

## 1. Problem Understanding

**Restated problem:** qode currently has two execution modes for prompt-generating commands (`qode plan refine`, `qode plan spec`, `qode start`, `qode review code`, `qode review security`, `qode review all`, `qode knowledge add-branch`): (a) interactive mode, which launches the Claude Code CLI and runs the generated prompt inside it; and (b) `--prompt-only` mode, which saves the prompt to a file and prints its path. IDE slash commands rely on `--prompt-only` — they call the command, then read the resulting file and execute it themselves. Since the IDE already provides the LLM runtime, the interactive dispatch is redundant and creates IDE lock-in.

**Chosen approach — stdout output + `--to-file` flag:**
The ticket describes this as the "even better" solution: affected commands write the generated prompt to `os.Stdout` by default. The LLM (via a slash command) runs the terminal command and reads its stdout as the prompt. A `--to-file` flag replaces `--prompt-only` for debugging — it saves the prompt to the existing file path and reports the path to `os.Stderr`.

**User need:** Any IDE with any LLM should be able to use qode's prompt generation without qode knowing about the AI runtime. qode generates; the IDE/LLM executes.

**Business value:** Removes IDE lock-in, eliminates the `dispatch` package (simplifying the binary), and makes the workflow IDE-agnostic.

**All ambiguities resolved:**
- `knowledge add-branch` **is** in scope — it uses `--prompt-only` and `dispatch.RunInteractive()`, so it must be updated the same way as all other affected commands.
- `qode review all` is confirmed removed.
- The TODO comment for the future `--force` flag belongs in `internal/cli/ide.go`, at the point where `ide.Setup()` is called in both `newIDESetupCmd` and `newIDESyncCmd`.
- VSCode IDE support (`internal/ide/vscode.go`, `SetupVSCode()`, `--vscode` flag) can be dropped: VSCode has no AI slash command mechanism, and qode's tasks/debug configs there are unrelated to prompt dispatch.

### Key technical decisions

1. **stdout as primary output.** All affected commands write the rendered prompt to `os.Stdout`. All non-prompt output (status messages, file-saved paths, errors) goes to `os.Stderr`. This keeps stdout clean for LLM consumption.

2. **stderr banner on stdout write.** When writing to stdout (i.e. no `--to-file`), print a one-line hint to `os.Stderr`:
   ```
   # Prompt written to stdout — pipe to your LLM or use --to-file to save.
   ```
   This makes direct terminal invocation understandable to developers inspecting output.

3. **`--to-file` flag.** Renders the template and saves to `.qode/branches/<branch>/.<type>-prompt.md` (existing path convention). Uses temp-file-then-rename for atomicity. Saves file path to `os.Stderr`. Template errors go to `os.Stderr`; process exits non-zero.

4. **Remove `qode review all`.** Deregister `newReviewAllCmd()` from the root command and delete the function. Verify it has no external references before deletion.

5. **Remove `internal/dispatch/` package.** After removing all `dispatch.RunInteractive()` calls from CLI files, verify with `grep -r '".*dispatch"' internal/` then delete `dispatch.go`, `claude.go`, `claude_test.go`. Run `go build ./...` to confirm.

6. **Drop VSCode IDE support.**
   - Delete `internal/ide/vscode.go`.
   - Remove the `cfg.IDE.VSCode.Enabled` block from `internal/ide/ide.go` (`SetupVSCode()` call).
   - Remove `--vscode` flag and `cfg.IDE.VSCode.Enabled` override from `newIDESetupCmd()` in `internal/cli/ide.go`.
   - Update docs and `qode.yaml` reference to note VSCode is no longer supported.

7. **`--force` TODO comment in `internal/cli/ide.go`.** Place this comment immediately above the `ide.Setup(root, cfg)` call in both `newIDESetupCmd` and `newIDESyncCmd`:
   ```go
   // TODO: add --force flag before beta to make overwriting opt-in; currently always overwrites.
   ```

8. **IDE slash command updates include `knowledge add-branch`.** The `qode-knowledge-add-branch` slash command in `claudeSlashCommands()` (claudecode.go) and `slashCommands()` (cursor.go) must be updated to call `qode knowledge add-branch` and use stdout output.

9. **Slash command instruction pattern.** Replace the current "run with `--prompt-only`, then read file" pattern with:
   ```
   Run: qode <command>
   Read the stdout output and use it as your prompt below.
   ```

### Patterns and conventions to follow

- Errors: `fmt.Errorf("context: %w", err)` — maintain throughout.
- Flag definitions: `cmd.Flags().BoolVar(&flag, "to-file", false, "save prompt to file instead of stdout")`.
- Functions max 50 lines; extract helpers if needed.
- Only the `--force` TODO comment is permitted in committed code; no other TODOs.

### Dependencies

## 3. Risk & Edge Cases

| Risk | Mitigation |
|---|---|
| Developer runs `qode plan refine` directly in terminal and sees raw prompt text | stderr banner: `# Prompt written to stdout — pipe to your LLM or use --to-file to save.` |
| Existing user repos have slash commands referencing `--prompt-only` | `qode ide sync` overwrites; document in release notes / migration guide |
| `qode review all` removal silently breaks user scripts | Grep README, docs, CI configs before deleting; note in changelog |
| stdout polluted by leftover `fmt.Print` calls in helper functions | Audit all `fmt.Print*` in affected files; redirect to `fmt.Fprint(os.Stderr, ...)` |
| `--to-file` writes partial file on template render error | Write to `os.CreateTemp`, rename on success, discard (and return error) on failure |
| dispatch package import left in one file causes build failure | Run `go build ./...` after each CLI file change; verify with grep before deleting package |
| VSCode removal breaks users who rely on `.vscode/tasks.json` generation | Document clearly in release notes; this is a breaking change for VSCode users |
| `knowledge add-branch` slash command not updated | Explicitly listed in task plan; judge against acceptance criteria |

**Security:** No new attack surface — removing a dispatch subprocess and a CLI flag is strictly subtractive.

**Performance:** Improved — no subprocess launch for normal invocation.
