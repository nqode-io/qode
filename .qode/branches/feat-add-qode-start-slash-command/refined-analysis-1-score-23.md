# Requirements Analysis: /qode-start Slash Command

## 1. Problem Understanding

### Restated Problem

The `qode start` CLI command generates an implementation prompt file but does not follow the same pattern as other commands in the system. Three specific issues exist:

1. **No slash command exists** for `qode start`, forcing users to leave their IDE and use the terminal — inconsistent with how `qode plan refine`, `qode plan spec`, `qode review code`, and `qode review security` work.
2. **Incomplete CLI behavior**: `qode start` writes the prompt file and copies to clipboard but does NOT use the dispatcher (`dispatch.Resolve()`) to automatically run prompts via Claude CLI when available, unlike equivalent commands.
3. **Stale user-facing message**: The output says `Paste into Cursor/Claude Code, or use: qode start --open` — but `--open` does not and should not exist.

### User Need and Business Value

Users should be able to generate and execute the implementation prompt entirely from within their IDE, matching the workflow of all other qode slash commands. This reduces friction by eliminating terminal context-switches and aligns the start command with the established developer workflow documented in CLAUDE.md.

### Ambiguities / Open Questions

1. **Dispatcher integration for `qode start`**: Other commands have `--prompt-only` to skip dispatch. Should `qode start` (the CLI) adopt the same flag pattern, invoking `dispatch.Resolve()` by default? The ticket says "should behave just like other commands," which implies yes — but this needs confirmation since the slash command will call the CLI with `--prompt-only`.
2. **VS Code equivalent behavior**: The ticket says "all IDEs have equivalent behavior." VS Code uses tasks (`tasks.json`) rather than slash commands. Should a VS Code task be added for `qode start`? Currently `plan refine`, `plan spec`, and reviews do not have VS Code tasks beyond initialization, so this may be out of scope.
3. **`qode start` slash command template structure**: Should it be a single-pass command (unlike refine/spec which have worker+judge passes), or does it have any multi-step aspect?

## 2. Technical Analysis

### Affected Components

| File | Change Required |
|------|----------------|
| `internal/cli/start.go` | Fix output message (remove `--open` mention); optionally add `--prompt-only` flag and dispatcher integration |
| `internal/ide/claudecode.go` | Add `qode-start` entry to `claudeSlashCommands()` |
| `internal/ide/cursor.go` | Add `qode-start` entry to `slashCommands()` |
| `internal/ide/vscode.go` | Add VS Code task for `qode start` (if in scope) |
| `.claude/commands/qode-start.md` | Generated Claude Code slash command file |
| `.cursor/commands/qode-start.mdc` | Generated Cursor slash command file |

### Key Technical Decisions

1. **Slash command template content**: The `/qode-start` slash command must:
   - Run `qode start --prompt-only` to generate `.qode/branches/$BRANCH/.start-prompt.md`
   - Instruct the AI agent (Claude Code / Cursor) to read that file and execute it as an implementation prompt directly in the session
   - This matches the pattern of `/qode-plan-refine` which calls `qode plan refine --prompt-only` and then reads/executes the resulting prompt

2. **`--prompt-only` flag for `qode start`**: The slash command pattern requires `qode start` to support `--prompt-only` so that:
   - The slash command invokes `qode start --prompt-only` (writes file + copies to clipboard, no dispatch)
   - `qode start` without the flag uses `dispatch.Resolve()` to auto-run via Claude CLI when available
   This matches how `plan.go` and `review.go` implement the same pattern.

3. **Removing `--open` reference**: In `internal/cli/start.go`, the final `fmt.Printf` or `render.Success()` call that prints the instruction message must be updated to remove the `qode start --open` mention.

4. **IDE generation**: The slash command files are auto-generated by `qode init` via `internal/ide/*.go`. Adding the new command to the generator functions ensures it is created for all new project initializations and updated on `qode init --update`.

### Patterns and Conventions to Follow

- **Slash command body pattern** (from `internal/ide/cursor.go` and `claudecode.go`):
  ```
  Run: qode start --prompt-only
  Read the prompt from: .qode/branches/$(git branch --show-current)/.start-prompt.md
  Execute the prompt as an implementation session.
  ```
- **`--prompt-only` flag** defined in `internal/cli/start.go` using `cmd.Flags().Bool("prompt-only", false, "...")`
- **Dispatcher call** pattern: `dispatch.Resolve(ctx, promptPath, ...)` — matches `plan.go:109` and `review.go`
- **Generated file naming**: `.claude/commands/qode-start.md` and `.cursor/commands/qode-start.mdc`

### Dependencies

- `internal/dispatch` package — already used by plan and review commands; no new dependencies
- `internal/ide` generator — already handles all other slash commands; adding `qode-start` is additive
- No external service dependencies

## 3. Risk & Edge Cases

### What Could Go Wrong

1. **Branch with no spec.md**: `qode start` already fails if `spec.md` is missing; the slash command should surface this error clearly rather than silently producing a broken prompt file.
2. **`--prompt-only` not added before removing `--open`**: If only the message is fixed but the flag isn't added, the slash command will invoke `qode start` without `--prompt-only` which would trigger dispatch — prompting Claude CLI to run the implementation immediately, which may not be desired at the slash command invocation point.
3. **Stale generated files**: Users who ran `qode init` before this change will not get `.claude/commands/qode-start.md` and `.cursor/commands/qode-start.mdc` until they re-run `qode init`. This is existing behavior for all command additions and is acceptable.
4. **VS Code parity**: If VS Code tasks are not added, the "all IDEs have equivalent behavior" requirement is only partially met. This is a minor gap since VS Code has no current slash-command equivalent for other workflow commands either.

### Security Considerations

- The generated slash command reads a local file path constructed from `git branch --show-current`. Branch names with special shell characters could cause issues in the shell interpolation used in `.mdc` files. Existing commands have the same surface and have not been flagged; this is low risk.
- No new network calls or privilege escalations introduced.

### Performance Implications

- None. Prompt generation is a local file write. No change to existing hot paths.

## 4. Completeness Check

### Explicit Acceptance Criteria

1. `/qode-start` slash command exists in Claude Code (`.claude/commands/qode-start.md`) and Cursor (`.cursor/commands/qode-start.mdc`).
2. Running `/qode-start` in Claude Code or Cursor executes the implementation prompt in the IDE without requiring terminal interaction.
3. `qode start` no longer mentions `--open` in its output.
4. `qode start` behavior matches `qode plan refine`, `qode review code`, etc. (supports `--prompt-only`; without the flag, dispatches via `dispatch.Resolve()`).
5. All IDEs with slash command support have equivalent `/qode-start` behavior.

### Implicit Requirements

- The slash command template must be added to the IDE generator so `qode init` produces it for all new projects.
- The existing `qode start` command must not break — the `--prompt-only` flag is additive and backward-compatible.
- The slash command instruction text must match the voice and style of other qode slash commands (e.g., `qode-plan-refine`).

### Explicitly Out of Scope

- A `--open` flag (ticket explicitly states it should not exist).
- Changes to the `qode start` prompt content or template (`internal/prompt/templates/start/`).
- New IDE integrations beyond Claude Code and Cursor slash commands.
- VS Code task integration (unless the team decides it's required for "all IDEs" parity — flagged as open question above).
