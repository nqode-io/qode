# Requirements Analysis: /qode-start Slash Command (Iteration 2)

## 1. Problem Understanding

### Restated Problem

The `qode start` CLI command generates an implementation prompt file but diverges from the established pattern of all other qode workflow commands. Three specific defects exist:

1. **No slash command**: Users must leave their IDE and run `qode start` from the terminal. Every other workflow command (`qode plan refine`, `qode plan spec`, `qode review code`, `qode review security`) has a corresponding IDE slash command.
2. **Missing dispatcher integration**: `qode start` writes the prompt file and copies to clipboard but never calls `dispatch.Resolve()` + `d.Run()`, so it cannot auto-execute via Claude CLI when available — unlike `plan.go` and `review.go`.
3. **Stale help text**: The final output message references `qode start --open`, a flag that does not and must not exist.

### User Need and Business Value

A developer mid-flow in their IDE should be able to type `/qode-start` and have the implementation prompt executed directly in their AI coding session — with no terminal context switch. This is the same UX contract all other qode slash commands provide and is the final link in the documented workflow: `plan spec` → `/qode-start` → implementation.

### Ambiguities Resolved

All three open questions from iteration 1 are now resolved by examining the codebase:

1. **Dispatcher integration**: Confirmed — `qode start` must adopt `--prompt-only` + `dispatch.Resolve()` / `d.Run()` exactly as `plan.go` and `review.go` do. Without `--prompt-only`, the CLI dispatches via Claude CLI when available; with `--prompt-only`, it writes the file and copies to clipboard only.

2. **VS Code parity**: Resolved — `plan refine` and `plan spec` have NO VS Code tasks in `internal/ide/vscode.go`. Only `review code`, `review security`, `fetch ticket`, and `check all` have VS Code tasks. Since `qode start` is an AI-interactive workflow command (like spec/refine), a VS Code task is **out of scope** by established precedent.

3. **Slash command template structure**: Confirmed — `qode start` is **single-pass** (like `qode-plan-spec`), not multi-pass. The slash command reads the generated `.start-prompt.md` and executes it as an implementation session with no intermediate scoring or judge pass.

### Key Technical Decisions

**1. `--prompt-only` flag and dispatch integration in `internal/cli/start.go`**

The required pattern (matching `plan.go:265-290` for `specDispatch()`):
```go
func startDispatch(ctx context.Context, branch string) error {
    promptPath, err := buildStartPrompt(branch)  // existing logic
    if err != nil {
        return err
    }
    d := dispatch.Resolve(ctx)
    if err := d.Run(promptPath); err != nil {
        if errors.Is(err, dispatch.ErrManualDispatch) {
            return startPromptOnly(branch)  // fallback: print file path + clipboard msg
        }
        return fmt.Errorf("start: %w", err)
    }
    return nil
}
```

When `d.Run()` returns `dispatch.ErrManualDispatch` (clipboard dispatcher path), the command falls back to `startPromptOnly()` which prints the file path and clipboard instruction — matching how `reviewDispatch()` calls `dispatch.ClipboardInstruction()` (`review.go:176`). Non-`ErrManualDispatch` errors are wrapped and propagated with `fmt.Errorf("start: %w", err)`.

**2. Slash command template — single-pass (matching `qode-plan-spec` pattern)**

Claude Code template content for `claudeSlashCommands()` in `internal/ide/claudecode.go`:
```
Run: qode start --prompt-only

Read the generated prompt from:
  .qode/branches/$BRANCH/.start-prompt.md

Execute the prompt as your implementation session.
```

The slash command does NOT have a worker+judge structure. It mirrors `qode-plan-spec.md` (single read-execute-save flow), not `qode-plan-refine.md` (worker+judge loop).

## 3. Risk & Edge Cases

### What Could Go Wrong

1. **Branch with no `spec.md`**: `qode start` already exits with an error if `spec.md` is missing. The slash command will surface this error in the IDE terminal output — no additional handling needed, but the error message should be user-friendly (existing behavior is acceptable).

2. **`dispatch.ErrManualDispatch` not handled**: If `ErrManualDispatch` is not caught before `fmt.Errorf` wrapping, the fallback to clipboard/prompt-only mode will not trigger and the user will see a raw error instead of a helpful instruction. The fix is the explicit `errors.Is()` check as shown in the pattern above.

3. **Implementation ordering**: `--prompt-only` must be implemented before updating the slash command generators. If generators reference `qode start --prompt-only` before the flag exists, the generated files will be correct but the CLI will reject the flag. This is a commit-ordering concern, not a code correctness concern.

4. **Stale generated files for existing users**: Users who ran `qode init` before this release will not have `.claude/commands/qode-start.md` or `.cursor/commands/qode-start.mdc`. They must re-run `qode init`. This is the same upgrade path for all previously-added slash commands — acceptable and consistent.

5. **`d.Run()` error with non-`ErrManualDispatch`**: If Claude CLI is present but returns a non-zero exit code (e.g., API rate limit, invalid session), `d.Run()` returns a regular error. This is wrapped and propagated to the user as a top-level command failure — consistent with how `plan.go` and `review.go` handle it.

### Security Considerations

- Branch names are interpolated into file paths via `git branch --show-current` in the `.mdc` template. A branch name with shell metacharacters (e.g., `$(cmd)`) could be interpreted by Cursor's shell expansion. This is an existing surface shared by all slash commands — not new risk, and no mitigation is added here.
- No new network calls, privilege escalations, or credential handling.

### Performance Implications

- Prompt generation is a local file read+write + Go template render. No hot paths affected.
