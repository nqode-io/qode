# Technical Specification — Reimplement `qode init`

**Branch:** reimplement-init-command
**Issue:** nqode-io/qode#29

---

## 1. Feature Overview

`qode init` is simplified from a detection-heavy, two-step setup process into a single self-contained command. It no longer auto-detects tech stacks or workspace topology, writes no project/layer config, and generates IDE slash commands as part of its single execution. The `qode ide` subcommand (setup + sync) is removed entirely.

After this change, running `qode init` in a project root:
1. Writes a minimal `qode.yaml` containing only `qode_version` and scoring/IDE defaults
2. Creates the `.qode/` directory structure
3. Copies embedded prompt templates to `.qode/prompts/` (user-editable overrides)
4. Generates Cursor (`.cursor/commands/*.mdc`) and Claude Code (`.claude/commands/*.md`) slash commands

**Business value:** reduces time-to-first-feature from two commands to one, eliminates unreliable stack detection that produces wrong test commands, and removes `.cursorrules/` files that compete with the user's own `CLAUDE.md`.

**Success criteria:**
- `qode init` completes successfully with zero arguments in any Go project directory
- `qode --help` does not list an `ide` subcommand
- No `.cursorrules/` directory is created
- All existing tests pass; new tests covering the init flow are added

## 5. Data Model Changes

### `qode.yaml` — minimal format after init

```yaml
qode_version: "0.1"
review:
  min_code_score: 10
  min_security_score: 8
scoring:
  strict: false
  rubrics:
    refine: { ... }
    review: { ... }
    security: { ... }
ide:
  cursor:
    enabled: true
  claude_code:
    enabled: true
knowledge:
  path: .qode/knowledge
```

Fields removed from `qode.yaml` output: `project` (name, topology, layers, test commands), `workspace`.

### Backward compatibility

Existing `qode.yaml` files containing `project:` or `workspace:` sections continue to deserialize without error — `gopkg.in/yaml.v3` ignores unknown fields. No migration is required. Users who re-run `qode init` will have their `qode.yaml` overwritten with the new minimal format (custom rubrics and scoring thresholds will be lost).

### Migration notes for breaking changes

1. **Local template overrides** (`qode/prompts/*.md.tmpl`) that use `{{range .Layers}}` or `{{.Layers}}` will panic when rendered after this change. `copyEmbeddedTemplates` overwrites these files on `qode init`, so users who re-run init will be updated automatically. Users with heavily customized local templates must remove `{{range .Layers}}` references manually.

2. **Stale `.cursorrules/` files** from prior qode runs are not removed. They remain valid and harmless but can be deleted manually.

## 8. Security Considerations

### No new attack surface

This change is purely a deletion and simplification. No new network calls, no new file paths derived from user input, no new config fields that flow into shell commands.

### `filepath.Base(root)` usage

`root` is the resolved project working directory (absolute path from `os.Getwd()` or the `--root` flag). `filepath.Base(root)` is used only in IDE command descriptions (markdown frontmatter text). It is never passed to `os.Exec`, `os/exec`, or any shell interpolation. No injection risk.

### Template execution

`TemplateProject.Name` flows into `text/template` execution. Go's `text/template` does not HTML-escape by default, but these templates produce markdown output (not HTML), and the name is sourced from the filesystem path — not user-supplied text. No sanitisation required.

### File write paths

All file writes are under `root` (resolved by `resolveRoot()`). No user-supplied paths are used for write destinations in the init flow. The `copyEmbeddedTemplates` function writes to `filepath.Join(root, config.QodeDir, "prompts", name+".md.tmpl")` where `name` comes from the embedded FS walk — not from user input.
