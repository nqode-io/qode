# Requirements Refinement Analysis — Iteration 2

## 1. Problem Understanding

`qode plan refine` is a CLI command with two execution paths:
- **Dispatch mode** (default): runs worker and judge prompts automatically via the Claude CLI and persists results
- **Prompt-only mode** (`--prompt-only`): writes prompts to disk for manual execution in an IDE

In both paths, `qode plan status` shows `0/25` for every iteration and the printed `Score: 0/25` after dispatch mode runs. The user cannot reliably gauge when to advance to `qode plan spec`, and the iterative improvement loop (worker → judge → score → repeat) is broken.

The `/qode-plan-refine` skill shares the same broken outcome via a different path: it runs the worker and saves the analysis but never invokes the judge at all.

**User need:** Know when their requirements analysis is good enough (≥ 25/25) to advance to spec generation.

**Business value:** The entire `qode plan refine` workflow is non-functional without accurate scoring. Users have no feedback signal and cannot trust the tool.

**Ambiguities resolved:** No open questions. All affected code and skill definitions are identified and confirmed by reading source.

There are **three distinct, independently-observable code bugs** plus **one skill gap**. All must be fixed.

### Bug 1 — Judge receives placeholder instead of real analysis

**File:** `internal/plan/refine.go:60-67` (`BuildRefinePromptWithOutput`)
**File:** `internal/cli/plan.go:173` (`refineDispatch`)

`BuildRefinePromptWithOutput` is called before the worker runs. The judge prompt is generated at that time with:

```go
scoreEngine.BuildJudgePrompt(
    "[Worker output will be pasted here by the user after running the worker prompt above]",
    scoring.RefineRubric,
)
```

The placeholder is intentional for `--prompt-only` mode. But in dispatch mode, `refineDispatch` later calls `d.Run(ctx, out.JudgePrompt, ...)` where `out.JudgePrompt` still contains the placeholder. The judge AI receives no real analysis and cannot produce a valid score.

**Root cause connection to `[DO NOT SCORE YOUR OWN OUTPUT]`:** The worker prompt template (`internal/prompt/templates/refine/base.md.tmpl:80`) explicitly tells the worker not to self-score. This means `scoring.ParseScore` on worker output is architecturally guaranteed to return 0 — the worker never includes "Total Score: X/25". The judge is the only valid source of a non-zero score.

**Fix:** After `savedAnalysis` is read in `refineDispatch`, build a fresh judge prompt using `string(savedAnalysis)` as the analysis content, then dispatch that fresh prompt.

### Bug 2 — Files are saved with `score=0` before the judge runs

**File:** `internal/cli/plan.go:175-178` (`refineDispatch`)

```go
result, _ := plan.ParseIterationFromOutput(root, branch, out.Iteration, string(savedAnalysis))
if judgeErr == nil && judgeOutput != "" {
    result = scoring.ParseScore(judgeOutput, scoring.RefineRubric)
}
```

`ParseIterationFromOutput` (`internal/plan/refine.go:131-147`) immediately:
1. Calls `scoring.ParseScore(analysisText, ...)` on worker output — returns 0 by design
2. Writes `refined-analysis-{N}-score-0.md` (score baked into filename)
3. Writes `refined-analysis.md` with header `<!-- qode:iteration=N score=0/25 -->`

After the judge runs (line 177), `result` is reassigned in memory but the files on disk are never updated. The filename `score-0` is permanent.

This bug is independent of Bug 1: even if the judge prompt were correct, the files would still be saved with `score=0` because persistence happens before judge output is available.

**Fix:** Run the judge, parse the score, then persist once with the correct score.

### Bug 3 — `LatestScore` mixes iteration number and score in a single variable

**File:** `internal/context/context.go:121-129`

```go
func (c *Context) LatestScore() int {
    best := 0
    for _, it := range c.Iterations {
        if it.Number > best {   // compares iteration index (1,2,3…) against best
            best = it.Score     // assigns score to best
        }
    }
    return best
}
```

`best` serves as both comparator (iteration number) and result (score). After the first iteration where `it.Score > 1`, `best` becomes a score value. All subsequent iterations whose `it.Number` is less than `best` (the score) are silently skipped.

**Concrete failure:** Iteration 1 scores 18. `best = 18`. Iteration 2: `2 > 18` → false, skipped. `LatestScore()` returns 18 permanently regardless of later iterations.

**No existing tests for `LatestScore`** — confirmed by reading `internal/context/context_test.go`.

**Fix:** Use two separate variables: one tracking the highest iteration number seen, one tracking the corresponding score.

### Skill Gap 4 — `/qode-plan-refine` never runs the judge

The skill runs `qode plan refine --prompt-only`, executes the worker prompt, saves the analysis, then stops. `.refine-judge-prompt.md` is written to disk but never executed. Score is never recorded.

After the code bugs are fixed (Bugs 1–3), the terminal path will correctly score in dispatch mode — but the skill path still leaves the user with no score.

**Fix:** After saving `refined-analysis.md`, the skill should:
1. Read `.refine-judge-prompt.md`
2. Replace the placeholder line (`[Worker output will be pasted here...]`) with the actual saved analysis content
3. Execute the modified judge prompt
4. Parse the score from the judge output
5. Rewrite `refined-analysis.md` with the correct `<!-- qode:iteration=N score=S/25 -->` header
6. Write `refined-analysis-{N}-score-{S}.md`

This mirrors exactly what the fixed terminal dispatch does, using the same file paths and header format, without requiring a new CLI command.

## 3. Risk & Edge Cases

| Risk | Detail | Mitigation |
|---|---|---|
| `savedAnalysis` empty after worker dispatch | Worker returned no output and file didn't pre-exist | Existing guard at `plan.go:155-162` covers this; fresh judge build only runs if `savedAnalysis` is non-empty |
| Judge dispatch fails (`judgeErr != nil`) | Claude CLI unavailable or returns error | Persist `result` with `TotalScore=0`; command does not crash |
| `BuildJudgePrompt` fails (template error) | Template engine error | Fall back to `out.JudgePrompt` (placeholder) so command does not abort |
| Existing `refined-analysis-1-score-0.md` from previous buggy runs | Old `score-0` files persist alongside new `score-S.md` files | `LatestScore` (after Bug 3 fix) finds the highest-numbered iteration's score; old files do not corrupt the result |
| Same iteration number with two files (`score-0.md` and `score-18.md`) | Possible if user re-runs after failed dispatch | `context.Load` adds both to `ctx.Iterations`; both share the same `Number`; `LatestScore` returns whichever appears last in the slice. Minor; acceptable for CLI tool. |
| `TwoPass = false` in config | No judge prompt generated | Existing `if out.JudgePrompt == "" \|\| judgePath == ""` guard at `plan.go:165` returns early. Score stays 0, which is correct for no-scoring config. |
| Skill: judge prompt placeholder replacement fails | Placeholder text not found in `.refine-judge-prompt.md` | Skill falls back to informing the user to run `qode plan refine` in the terminal |
| Skill: iteration number detection | Skill needs to know which iteration it's on to write the correct filename | Read `<!-- qode:iteration=N -->` header from existing `refined-analysis.md` if present, otherwise default to 1 |

func SaveIterationResult(root, branch string, iteration int, analysisText string, result scoring.Result) error {
    branchDir := filepath.Join(root, config.QodeDir, "branches", branch)
    iterFile := filepath.Join(branchDir, fmt.Sprintf("refined-analysis-%d-score-%d.md", iteration, result.TotalScore))

    }

    header := buildAnalysisHeader(iteration, result)
    return os.WriteFile(latestFile, []byte(header+analysisText), 0644)
}
