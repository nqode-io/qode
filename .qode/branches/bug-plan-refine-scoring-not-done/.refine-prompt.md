# Requirements Refinement Analysis

## 1. Problem Understanding

`qode plan refine` supports two execution paths:
- **Dispatch mode** (default): runs worker and judge prompts automatically via the Claude CLI
- **Prompt-only mode** (`--prompt-only`): writes prompts to disk for manual execution

In both paths, after the command completes the score shown (and persisted) is always `0/25`. The user cannot progress to `qode plan spec` with confidence, nor can `qode plan status` surface useful progress.

There are **three distinct bugs** that together cause the symptom. All require a fix; each is independently observable.

---

## 2. Technical Analysis

### Bug 1 — Judge receives a placeholder string, not the real analysis

**File:** `internal/plan/refine.go`, `BuildRefinePromptWithOutput` (line 60–67)
**File:** `internal/cli/plan.go`, `refineDispatch` (line 173)

`BuildRefinePromptWithOutput` is called **before the worker runs**. The judge prompt is generated at that point with the literal string:

```
"[Worker output will be pasted here by the user after running the worker prompt above]"
```

passed as `workerOutput` to `scoring.Engine.BuildJudgePrompt`. The placeholder is appropriate for `--prompt-only` mode (where the user pastes content manually), but in dispatch mode the judge is later invoked with `out.JudgePrompt` — which still contains the placeholder. The judge AI sees no real analysis, so it cannot produce a valid score.

**Affected code path:**
```
refineDispatch (plan.go:173)
  d.Run(ctx, out.JudgePrompt, ...)   ← out.JudgePrompt has placeholder
```

**Fix:** After `savedAnalysis` is read in `refineDispatch`, build a fresh judge prompt using the actual analysis content before running the judge.

---

### Bug 2 — `ParseIterationFromOutput` is called before the judge runs, saving `score=0`

**File:** `internal/cli/plan.go`, `refineDispatch` (lines 175–178)

```go
result, _ := plan.ParseIterationFromOutput(root, branch, out.Iteration, string(savedAnalysis))
if judgeErr == nil && judgeOutput != "" {
    result = scoring.ParseScore(judgeOutput, scoring.RefineRubric)
}
```

`ParseIterationFromOutput` saves two artifacts:
1. `refined-analysis-{N}-score-{S}.md` — numbered iteration file (filename encodes the score)
2. `refined-analysis.md` — the canonical latest file with header `<!-- qode:iteration=N score=S/25 -->`

It parses the score by calling `scoring.ParseScore` on the **worker analysis text**, which never contains a "Total Score: X/25" line. So `result.TotalScore` is `0` at save time.

Even if the judge subsequently returns a real score (which it can't due to Bug 1, but hypothetically), `result` is reassigned in memory but the **files are never re-saved** with the corrected score. The filesystem always shows `score-0`.

**Fix:** Collect the judge score first, then call `ParseIterationFromOutput` (or a new variant) once, with the correct score.

---

### Bug 3 — `LatestScore` compares iteration number against accumulated score variable

**File:** `internal/context/context.go`, `LatestScore` (lines 121–129)

```go
func (c *Context) LatestScore() int {
    best := 0
    for _, it := range c.Iterations {
        if it.Number > best {   // BUG: it.Number is iteration index (1,2,3…)
            best = it.Score     //       but best holds a score (0-25)
        }
    }
    return best
}
```

The variable `best` is used to track two different things: it starts as an iteration-number comparator (`it.Number > best`) but is assigned the score (`best = it.Score`). After the first iteration with any score > 1, subsequent iterations whose index is less than the current score are skipped.

Example: iteration 1 scores 18 → `best = 18`. Iteration 2: `2 > 18` is false → skipped. Iteration 3: `3 > 18` is false → skipped. Result: `LatestScore()` permanently returns the score from iteration 1 regardless of later iterations.

The intent is to return the score of the **highest-numbered** (most recent) iteration.

**Fix:** Track iteration number and score in separate variables.

## 4. Completeness Check

### Acceptance Criteria

1. After `qode plan refine` completes in dispatch mode, the printed `Score: N/25` reflects the judge's actual assessment.
2. After dispatch completes, `refined-analysis-{N}-score-{S}.md` has the correct score `S` (not 0) in the filename.
3. After dispatch completes, `refined-analysis.md` has `<!-- qode:iteration=N score=S/25 -->` with the correct score.
4. `qode plan status` displays the correct score for each iteration.
5. `LatestScore()` returns the score of the most recently numbered iteration, even when multiple iterations exist.
6. `--prompt-only` mode is unaffected: judge prompt file still contains the placeholder (for manual use).

### Implicit Requirements

- The fix must not change the public API of `plan.BuildRefinePromptWithOutput` or `plan.SaveIterationFiles` — other callers (tests) may depend on them.
- `refineDispatch` needs access to the prompt engine to rebuild the judge prompt; `engine` must be threaded through to it.
- The numbered iteration file and the `refined-analysis.md` header must be written in a single `ParseIterationFromOutput` call after the correct score is known.

### Out of Scope

- Automated looping (running worker+judge until score reaches 25 without user intervention)
- `MaxRefineIterations` enforcement (exists in config but is intentionally not enforced)
- `--prompt-only` judge-score capture (the score is not auto-persisted in that flow by design)

func SaveIterationResult(root, branch string, iteration int, analysisText string, result scoring.Result) error {
    branchDir := filepath.Join(root, config.QodeDir, "branches", branch)
