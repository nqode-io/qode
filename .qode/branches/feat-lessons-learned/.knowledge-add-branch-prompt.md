# Feature: /qode-ticket-fetch Slash Command

---

## 1. Problem Understanding

**Restated Problem**: Users currently must switch from their IDE to a terminal to run `qode ticket fetch <url>` when they want to pull a ticket (GitHub issue, Jira, Linear, Azure DevOps) into their branch context. This breaks IDE flow. The request is to create a Claude Code slash command `/qode-ticket-fetch <url>` so users can trigger the ticket fetch without leaving the IDE.

**User Need**: Reduce context switching during the early workflow phase (after `qode branch create`, before `/qode-plan-refine`). The ticket fetch populates `.qode/branches/<branch>/context/ticket.md`, which is the primary input for the requirements refinement process.

**Business Value**: Keeps the entire qode workflow executable from within Claude Code without needing a terminal, consistent with the IDE-first developer experience provided by `/qode-plan-refine`, `/qode-plan-spec`, `/qode-review-code`, and `/qode-review-security`.

**Ambiguities / Open Questions**:
- Q1: Should the slash command run `qode ticket fetch <url>` directly (Claude executes it as a bash command), or should it use a `--prompt-only` pattern to generate an intermediate prompt file? Answer: Direct execution is correct. Unlike plan/review commands, ticket fetch is a deterministic API call with no AI analysis step. The `--prompt-only` pattern exists only for AI-assisted workflows.
- Q2: Should the command read and summarize the saved ticket for the user after fetching? This is a UX decision. Other slash commands include a "next step" instruction — recommending the same pattern here.
- Q3: Is the `$ARGUMENTS` substitution mechanism in Claude Code slash command `.md` files confirmed available? Yes — Claude Code supports `$ARGUMENTS` in command files, replaced by whatever the user types after the slash command name.

### Key Technical Decisions

**Decision 1 — Command content pattern**: Use the `!` prefix in the slash command file to execute the bash command directly, bypassing Claude entirely. The entire file content is:
```
!qode ticket fetch $ARGUMENTS
```
This is correct because ticket fetching is a deterministic API call — no AI interpretation step is needed or wanted.

**Decision 2 — Post-fetch behavior**: With direct `!` execution, there is no Claude-mediated post-fetch summary. The CLI output (`Fetched ticket: <title>` + `Saved to: <path>`) is surfaced directly to the user. This is sufficient.

**Decision 3 — URL parameter passing**: The slash command uses `$ARGUMENTS` for the URL. Users invoke it as `/qode-ticket-fetch https://github.com/owner/repo/issues/123`. The URL is passed verbatim to the CLI, which validates it via `ticket.DetectProvider()`.

**Decision 4 — Slash command file naming**: Must be `qode-ticket-fetch` (kebab-case), consistent with: `qode-plan-refine`, `qode-plan-spec`, `qode-review-code`, `qode-review-security`.

## 3. Risk & Edge Cases

### What Could Go Wrong

| Scenario | Risk | Mitigation |
|---|---|---|
| User forgets to provide URL | `$ARGUMENTS` is empty; `qode ticket fetch` receives no args and fails with cobra's `ExactArgs(1)` error | Claude should validate that arguments are non-empty before running and prompt the user if missing |
| Malformed URL | `ticket.DetectProvider()` returns error "no provider found for URL" | Already handled in CLI; error propagates to Claude which should display it clearly |
| Missing auth token (e.g., `GITHUB_TOKEN` not set) | Provider fetch fails with auth error | Already handled in GitHub provider; error propagates. Slash command should instruct Claude to check token setup if auth error occurs |
| Network failure | Provider HTTP call fails | Already handled; error propagates from `provider.Fetch()` |
| Not in a git repo or no branch | `git.CurrentBranch()` fails | Already handled; error propagates |
| Ticket already fetched (ticket.md exists) | File is silently overwritten by `os.WriteFile()` | This is acceptable behavior. Refetch = update. No change needed. |
| Branch context dir doesn't exist | `os.MkdirAll()` creates it | Already handled in `newTicketFetchCmd()` |
| Large ticket body | No size limits imposed | Acceptable — ticket content is plain text, no memory risk |

### Security Considerations

- **URL injection**: The URL from `$ARGUMENTS` is passed to `qode ticket fetch`, which is a trusted CLI binary. The URL is validated by `ticket.DetectProvider()` which checks URL structure before any HTTP request. No shell interpolation risk since Claude executes the command as a structured subprocess, not via shell expansion.
- **Credential exposure**: `GITHUB_TOKEN` and other tokens are environment variables, never written to disk. The ticket content written to `ticket.md` comes from the provider API response and is treated as plain text. No sanitization needed beyond what the provider already does (returning `Ticket{Title, Body}`).
- **Path traversal**: The output path is constructed by `filepath.Join(root, config.QodeDir, "branches", branch, "context", "ticket.md")` where `branch` comes from `git.CurrentBranch()`, not from user input. No traversal risk.

### Performance Implications

- Single HTTP API call per invocation. GitHub API has rate limits (60 req/hr unauthenticated, 5000/hr authenticated). No caching implemented — acceptable for interactive CLI use.
- File write is O(ticket body size) — negligible.

# Requirements Analysis: GitHub Issues Integration

**Branch:** feat-github-issue-support
**Date:** 2026-03-03

---

## 1. Problem Understanding

### Restated Problem

`qode ticket fetch <url>` supports three providers today: Jira, Azure DevOps, and Linear (registered in `internal/ticket/ticket.go:init()` lines 28–33). GitHub Issues is the most common issue tracker for open-source projects and small teams, yet there is no way to use a `github.com` URL with the command. This ticket adds GitHub Issues as a fourth provider, following the identical extension pattern used by the existing three.

### User Need & Business Value

- **User need:** `qode ticket fetch https://github.com/owner/repo/issues/42` saves the issue title and body to `.qode/branches/{branch}/context/ticket.md` in the same `# {Title}\n\n{Body}\n` format written by `internal/cli/ticket.go:59`: `fmt.Sprintf("# %s\n\n%s\n", t.Title, t.Body)`.
- **Business value:** Removes friction for GitHub-native teams and open-source contributors; makes qode usable without any paid third-party ticketing tool.

### Open Questions (assumptions documented)

1. **Private repos:** Assumed yes — a token with `repo` scope or fine-grained `Issues: Read` permission grants access. Without a token, public repos still work (GitHub REST API allows unauthenticated reads at 60 req/hr).
2. **Comments vs description:** Assumed description only (`body` field), matching the depth returned by Jira, Azure DevOps, and Linear.
3. **Pull Requests:** Out of scope — ticket says "issues". `CanHandle` must reject `/pull/` paths.
4. **GitHub Enterprise Server:** Out of scope — `CanHandle` only matches the `github.com` host.

### Key Technical Decisions

#### A. GitHub REST API v3 (not GraphQL)

Endpoint: `GET https://api.github.com/repos/{owner}/{repo}/issues/{number}`

Linear uses GraphQL because Linear's REST API is incomplete. GitHub's REST API is first-class. Using REST keeps the provider consistent with `azuredevops.go` and `jira.go` and avoids a new dependency pattern. No new imports beyond stdlib.

Required headers per GitHub best practices:
- `Accept: application/vnd.github+json`
- `X-GitHub-Api-Version: 2022-11-28`
- `Authorization: Bearer {token}` (omitted when token is empty)

#### B. GITHUB_TOKEN is optional (unlike Jira/ADO)

Jira and Azure DevOps check the env var before making any HTTP request and return an error if missing (`jira.go:32–35`, `azuredevops.go:28–31`). GitHub cannot follow this pattern because public repos are accessible without authentication.

Correct approach: always attempt the request. Conditionally set the `Authorization` header if `GITHUB_TOKEN` is set. On a 401 or 403 response, return an actionable error that mentions `GITHUB_TOKEN`. On a 404, return an error that mentions the possibility of a private repo requiring the token.

Error format follows the existing convention from `jira.go:33` and `azuredevops.go:30`:
```
GITHUB_TOKEN environment variable not set
Set it with: export GITHUB_TOKEN=your-token
```

#### C. URL Parsing

GitHub issue URL structure: `https://github.com/{owner}/{repo}/issues/{number}`

- `hostContains(rawURL, "github.com")` — reuse the helper defined in `ticket.go:53–59`
- Parse path with `url.Parse`, use `path.Clean` to strip trailing slash, split by `/`, validate: `len(segments) >= 5`, `segments[3] == "issues"`, `segments[4]` is a valid decimal string
- `CanHandle` checks both host and path; returns `false` for `/pull/` paths and for `github.com/owner/repo` (no issue number)

#### D. Response Parsing

JSON response fields of interest:
```json
{ "number": 42, "title": "Fix the bug", "body": "Markdown text or null", "html_url": "..." }
```

`body` is already Markdown — no `stripHTML` call needed (contrast with `azuredevops.go:66` which calls `stripHTML(result.Fields.Description)`). `body` can be JSON `null` for issues with no description; decode into `Body *string` and default to empty string when nil.

`Ticket.ID` should be the issue number as a decimal string (e.g., `"42"`), consistent with `azuredevops.go:64`: `fmt.Sprintf("%d", result.ID)`. Use `strconv.Itoa(issue.Number)`.

`Ticket.URL` should be `rawURL` (the original input URL), consistent with all three existing providers which set `URL: rawURL`.

#### E. Testability — apiBase injection

Existing providers have no HTTP tests because they hardcode production URLs and provide no injection mechanism. This provider must be testable with `httptest.NewServer`.

`GitHubProvider` will have one unexported field:

```go
type GitHubProvider struct {
    apiBase string // empty → "https://api.github.com"
}
```

Production code (in `ticket.go` init): `&GitHubProvider{}` — zero value, uses default.
Tests: `&GitHubProvider{apiBase: server.URL}` — points at mock server.

This is the minimal deviation from the zero-value pattern used by other providers (`JiraProvider{}`, `AzureDevOpsProvider{}`, `LinearProvider{}` — all empty structs) while enabling reliable testing.

#### F. Function decomposition (50-line constraint)

- `(p *GitHubProvider) Name() string` — 1 line
- `(p *GitHubProvider) CanHandle(rawURL string) bool` — host + path check, ~10 lines
- `parseGitHubURL(rawURL string) (owner, repo, number string, err error)` — URL decomposition, ~20 lines
- `(p *GitHubProvider) fetchIssue(owner, repo, number string) (*githubIssue, error)` — HTTP + JSON, ~35 lines
- `(p *GitHubProvider) Fetch(rawURL string) (*Ticket, error)` — orchestrator, ~15 lines

All well within the 50-line limit.

#### G. Required imports for `github.go`

All stdlib — no new third-party dependencies:
- `encoding/json` — JSON decode
- `fmt` — error formatting, Sprintf for API URL
- `io` — `io.ReadAll` for non-200 error body (same as `jira.go:51`, `azuredevops.go:48`)
- `net/http` — HTTP client
- `net/url` — URL parsing in `parseGitHubURL`
- `os` — `os.Getenv("GITHUB_TOKEN")`
- `path` — `path.Clean` to normalize trailing slash
- `strconv` — `strconv.Atoi` in `parseGitHubURL`, `strconv.Itoa` in `Fetch`

### Patterns to Follow (verified against actual code)

- `hostContains()` from `ticket.go:53–59` — call in `CanHandle`.
- HTTP flow from `azuredevops.go:34–68`: `http.NewRequest` → set headers → `http.DefaultClient.Do` → check `resp.StatusCode != 200` → `io.ReadAll` for error body → `json.NewDecoder(resp.Body).Decode`.
- Error prefix for fetch errors from `jira.go:46`: `fmt.Errorf("fetching Jira issue: %w", err)` → use `"fetching GitHub issue: %w"`.

- Generic API error from `jira.go:52`: `fmt.Errorf("Jira API returned %d: %s", resp.StatusCode, string(body))` → use `"GitHub API returned %d: %s"`.
- Token env-var error from `jira.go:33`: `fmt.Errorf("JIRA_API_TOKEN environment variable not set\nSet it with: export JIRA_API_TOKEN=your-token")`.
- `Ticket.URL = rawURL` — all three existing providers set this to the input URL, not the API response URL.
- `Ticket.ID` — string representation of identifier: Jira uses key string, ADO uses `fmt.Sprintf("%d", result.ID)`, Linear uses identifier string.

### Dependencies on Other Features

## 3. Risk & Edge Cases

| Risk | Handling |
|---|---|
| `body` is JSON `null` | Decode into `Body *string`; use `""` when nil in `Fetch` |
| 404 without token (private repo) | `"GitHub issue not found — if this is a private repository, set GITHUB_TOKEN\nSet it with: export GITHUB_TOKEN=your-token"` |
| 401 with bad/expired token | `"GitHub API returned 401 — check GITHUB_TOKEN\nSet it with: export GITHUB_TOKEN=your-token"` |
| 403 rate limit (no token, public repo) | Same pattern as 401 — `"GitHub API returned 403 — check GITHUB_TOKEN\nSet it with: export GITHUB_TOKEN=your-token"` |
| Issue number not an integer | `strconv.Atoi` in `parseGitHubURL`; return `fmt.Errorf("invalid GitHub issue number: %q", segments[4])` |
| Trailing slash in URL | `path.Clean` before splitting — strips trailing slash |
| PR URL (`/pull/`) passed | `CanHandle` checks `segments[3] == "issues"`; returns `false` for PR URLs |
| `owner`/`repo` with special path characters | Both values come from `url.Parse` path segments; apply `url.PathEscape` before embedding in API URL |
| Empty `owner` or `repo` | Validate non-empty in `parseGitHubURL`; return `fmt.Errorf("could not parse GitHub URL: missing owner or repo: %s", rawURL)` |
| `http.NewRequest` error | Return it directly — identical to all three existing providers |
| `http.DefaultClient.Do` network error | Return `fmt.Errorf("fetching GitHub issue: %w", err)` — matches `jira.go:46` pattern |
| Non-200/non-404/non-401/non-403 status | `io.ReadAll(resp.Body)` → `fmt.Errorf("GitHub API returned %d: %s", ...)` — matches `jira.go:52` |

### Security Considerations

- **No token logging:** The `Authorization` header value must never appear in error messages or logs. All existing providers maintain this; match the pattern. The `fetchIssue` function reads `GITHUB_TOKEN` but only interpolates it into the request header, never into returned errors.
- **URL injection:** `owner`, `repo`, and `number` are interpolated into the API URL path. `url.PathEscape` each segment before interpolation to prevent path traversal. `number` is already validated as a decimal integer by `strconv.Atoi` in `parseGitHubURL`, so it is safe without escaping, but `owner` and `repo` must be escaped.
- **Response size:** No limit imposed by existing providers; maintain consistency. Note as future hardening.
- **Token scope:** Document minimum: classic PAT with `repo` scope (private repos) or no scope (public); fine-grained PAT with `Issues: Read`.

### Performance

Single HTTP GET per invocation. Identical to existing providers.

# Requirements Refinement — bug-ticket-fetch-does-not-apply-dot-env

## 1. Problem Understanding

**Restated Problem:**
When a user runs the `/qode-ticket-fetch` slash command from within a VS Code or other IDE extension, the underlying `qode ticket fetch <url>` CLI command cannot access secrets like `GITHUB_TOKEN`. This is because the CLI reads tokens via `os.Getenv()` but never loads a `.env` file — so tokens only exist if the user explicitly exports them in a shell session. IDE-launched processes typically do not inherit the user's interactive shell environment.

**User Need & Business Value:**
Developers should be able to store API tokens in a `.env` file at the project root and have all `qode` CLI commands transparently pick them up. This eliminates the need for manual shell exports and makes the tool usable from IDEs without extra setup.

**Ambiguities:**
- Should `.env` loading be applied globally at program startup, or per-command?
- Should `.env` override existing env vars, or only fill in missing ones? (convention: `.env` should NOT override existing system/shell env vars)
- Should we search for `.env` in the project root (resolved via `qode.yaml` location), the working directory, or both?
- Are there other CLI subcommands that also use `os.Getenv()` directly that need the same fix? (Quick scan: only ticket providers do today, but the ticket says "apply to all slash commands")

## 2. Technical Analysis

**Affected Components:**

| Component | Path | Role |
|-----------|------|------|
| Main entry point | `cmd/qode/main.go` | Best single place to load `.env` early |
| Ticket fetch command | `internal/cli/ticket.go:23-69` | Calls provider which uses `os.Getenv()` |

2. **`.env` file location resolution:** The project root is determined by walking up from `cwd` until a `qode.yaml` is found (see `internal/config/config.go`). Load `.env` from that same directory. If no project root is found (e.g., `qode init`), fall back to `cwd`.

3. **Non-override semantics:** Standard `.env` convention: only set an env var if it is not already set in the environment. This preserves CI/shell-exported values.

4. **Library choice:** Use `github.com/joho/godotenv` — the canonical Go `.env` library. It is not currently a dependency, so it must be added. 5. **Silent failure when `.env` missing:** If no `.env` file exists, proceed silently. This is not an error.

**Patterns to Follow:**
- Config loading is done in the `internal/config` package. `.env` loading should live nearby or in a new `internal/env` package, keeping `main.go` thin.
- Error handling is explicit throughout; `.env` parse errors should log a warning but not fatal-exit.

# Requirements Refinement Analysis — bug-plan-refine-not-reentrant

## 1. Problem Understanding

### Restatement

The `/qode-plan-refine` slash command (defined in `internal/ide/claudecode.go`) orchestrates the refinement workflow by:

## 2. Technical Analysis

### Root Cause

**File:** `internal/context/context.go`, `Load()` function (lines 77–94)

Iteration tracking uses a glob: `refined-analysis-*.md`. This only matches *numbered* iteration files (e.g., `refined-analysis-1-score-22.md`).

**File:** `internal/plan/refine.go`, `BuildRefinePromptWithOutput()` (lines 30–33)

```go

```

The iteration number is derived from `len(ctx.Iterations)`. If no numbered files exist, iteration is always `1`.

**The gap:** The slash command tells Claude to write directly to `refined-analysis.md`. This file contains an HTML comment header with the iteration number:

```

```

...but `context.Load()` never reads this header back. The numbered file (`refined-analysis-1-score-22.md`) is only created when `ParseIterationFromOutput()` is called — which only happens in the non-`--prompt-only` flow (i.e., terminal `qode plan refine` with AI dispatch).

### Key Technical Decisions

**1. How to pass the prompt in interactive mode:**

The current code uses `cmd.Stdin = strings.NewReader(prompt)`. In interactive mode, `stdin` must be `os.Stdin` for terminal interaction — so we cannot pass the prompt via stdin.

Verified via `claude --help`:
- `--file <specs...>` means "file resources to download at startup" (format: `file_id:relative_path`) — **not** a prompt file path. This flag is not suitable.
- `prompt` is a positional argument ("Your prompt").

**Chosen approach:** Write the prompt to a temp file, pass a short read-and-execute instruction as the positional argument:
```
claude --allowedTools "..." --model sonnet "Read and execute the instructions in /tmp/qode-prompt-XXXX.md"
```
This avoids ARG_MAX limits (review prompts with large diffs can be large), matches the existing IDE slash command pattern (which also instructs Claude to read a file), and requires no new CLI flags.

**2. New method `RunInteractive` on `claudeCLI`:**
Add to `internal/dispatch/claude.go` — not to the `Dispatcher` interface (clipboard has no interactive analog). Expose a package-level `RunInteractive()` function.

**3. Removing `--print` and `--output-format`:**
These flags activate batch mode. Interactive mode omits them. `--allowedTools` and `--model` are preserved.

**4. Wiring stdio:**
`cmd.Stdin = os.Stdin`, `cmd.Stdout = os.Stdout`, `cmd.Stderr = os.Stderr`.

**5. Return value:**
`RunInteractive` returns only `error` (no string — Claude writes output files directly).

**6. Context/timeout:**
Interactive sessions use `context.Background()` with no deadline. Rely on `cmd.Wait()` for natural process termination.

**7. TTY detection — stdlib only, no new dependency:**
`go.mod` has no `golang.org/x/term`. Use `os.Stdin.Stat()`:
```go
func isTTY() bool {
    fi, err := os.Stdin.Stat()
    if err != nil {
        return false
    }
    return (fi.Mode() & os.ModeCharDevice) != 0
}
```
If not a TTY, `RunInteractive` falls back to `c.Run(ctx, prompt, opts)` (batch mode).

**8. `CLAUDECODE` env filter:** Preserved in `RunInteractive` via existing `filterEnv()`.

### Patterns to Follow

- Follow existing `exec.CommandContext` style in `internal/dispatch/claude.go`.
- Keep `Dispatcher` interface unchanged.
- Expose a package-level `RunInteractive(ctx, prompt, opts)` parallel to `Resolve().Run(...)`.
- Call sites (`review.go`, `plan.go`, `start.go`) use `dispatch.RunInteractive(...)` in place of `d.Run(...)`.

### Dependencies

- No new external dependencies — TTY detection via stdlib `os.Stdin.Stat()`.
- `claude` binary availability already checked via `claudeCLI.Available()`.

## 3. Risk & Edge Cases

1. **ARG_MAX for prompt size:** Review prompts with large diffs could exceed command-line length limits (~1MB on macOS). **Mitigation:** Write prompt to temp file; pass only a short read-instruction as the positional argument. Temp file is cleaned up via `defer os.Remove(f.Name())`.

2. **Non-TTY environments (CI, pipes):** `os.Stdin` not a TTY — interactive mode undefined behaviour. **Mitigation:** `isTTY()` check using `os.Stdin.Stat()`; fall back to batch mode if not a TTY. No new dependency needed.

3. **Signal propagation (Ctrl+C):** Must terminate the `claude` subprocess cleanly. `exec.Command` without custom `SysProcAttr` propagates SIGINT from parent to child on macOS/Linux. `cmd.Wait()` returns the exit error; callers propagate it up. No extra signal handling needed.

4. **`qode review all` — first session error or abort:** If code review exits non-zero, `runReview("security")` must not start. Existing `runReview("code") → runReview("security")` with error propagation handles this correctly — no extra change needed.

5. **Score extraction after interactive run:** `reviewDispatch()` reads the score from the file Claude wrote after the session ends. Since `RunInteractive` blocks until Claude exits, the file is fully written before score extraction. No change to scoring logic.

6. **Clipboard fallback:** `clipboardDispatcher` has no interactive analog — it stays unchanged. `RunInteractive` falls back to clipboard's `Run()` when `claudeCLI` is unavailable.

7. **Temp file cleanup on error:** If `f.WriteString` or `f.Close()` fails before `cmd.Run()`, `defer os.Remove(f.Name())` still executes. If `cmd.Run()` panics, defer also fires. Safe.

8. **No zombie processes:** `cmd.Wait()` (called by `cmd.Run()`) reaps the process on exit.

### Security Considerations

- No user input interpolated into shell invocation — `exec.Command` used (not shell). No injection risk.
- Temp file created with `os.CreateTemp("", "qode-prompt-*.md")` in system temp dir with `0600` permissions by default.
- `CLAUDECODE` env stripping preserved.

### Performance

- No regression — interactive mode is a direct process handoff with lower overhead than buffer capture.

```go
// RunInteractive runs claude as a foreground interactive process, wiring
// stdin/stdout/stderr to the current terminal. Falls back to batch mode
// when stdin is not a terminal (e.g., CI environments).
func (c *claudeCLI) RunInteractive(ctx context.Context, prompt string, opts Options) error {
    if !isTTY() {
        _, err := c.Run(ctx, prompt, opts)
        return err
    }

    // Write prompt to temp file: stdin must be available for terminal interaction.
    f, err := os.CreateTemp("", "qode-prompt-*.md")
    if err != nil {
        return fmt.Errorf("create temp prompt file: %w", err)
    }
    defer os.Remove(f.Name())
    if _, err := f.WriteString(prompt); err != nil {
        return fmt.Errorf("write prompt: %w", err)
    }
    if err := f.Close(); err != nil {
        return fmt.Errorf("close prompt file: %w", err)
    }

    // Pass a read-and-execute instruction as the positional prompt argument.
    // This avoids ARG_MAX limits for large prompts.
    readInstruction := fmt.Sprintf("Read and execute the instructions in %s", f.Name())

    cmd := exec.CommandContext(ctx, c.binaryPath,
        "--allowedTools", "Read,Write,Glob,Grep",
        "--model", "sonnet",
        readInstruction,
    )
    cmd.Stdin = os.Stdin
    cmd.Stdout = os.Stdout
    cmd.Stderr = os.Stderr
    cmd.Env = filterEnv(os.Environ(), "CLAUDECODE")
    if opts.WorkingDir != "" {
        cmd.Dir = opts.WorkingDir
    }
    return cmd.Run()
}

// RunInteractive is a package-level convenience that resolves the best
// available dispatcher and runs it interactively. Falls back to clipboard
// if claude CLI is not available.
func RunInteractive(ctx context.Context, prompt string, opts Options) error {
    if d := newClaudeCLI(); d.Available() {
        return d.RunInteractive(ctx, prompt, opts)
    }
    d := &clipboardDispatcher{}
    _, err := d.Run(ctx, prompt, opts)
    return err
}
```

## 8. Security Considerations

### Token Handling

- `GITHUB_TOKEN` is read via `os.Getenv` and written only to the `Authorization` request header. It must **never** appear in error messages, log output, or returned error strings. All error paths in `fetchIssue` are constructed before the token is read or contain only the HTTP status code and response body — never the token value.
- Document minimum required token scope: classic PAT with `repo` scope for private repos (no scope needed for public); fine-grained PAT with `Issues: Read`.

### Input Validation

- `owner` and `repo` path segments are extracted from `url.Parse` and passed through `url.PathEscape` before embedding in the API URL to prevent path traversal.
- `number` is validated as a decimal integer via `strconv.Atoi` in `parseGitHubURL`. Since it is validated as pure digits, it is safe to embed in the URL without additional escaping, but `url.PathEscape` is harmless and may be applied for consistency.
- `CanHandle` validates path structure (length ≥ 5 segments, `segments[3] == "issues"`, numeric `segments[4]`) before `Fetch` is called, providing defense in depth.

### Authentication / Authorisation

- No auth state is persisted or cached. Each `Fetch` call reads `GITHUB_TOKEN` fresh from the environment.
- Unauthenticated requests to public repos are allowed by design. The provider does not fail-fast on a missing token (unlike Jira and Azure DevOps providers which error immediately if their required env vars are absent).

### Data Sensitivity

- Issue title and body are written to `.qode/branches/{branch}/context/ticket.md`, which is a local developer file. No data is transmitted beyond the single outbound GitHub API request.
- Future hardening: impose a response body size limit (e.g., 1 MB) consistent with GitHub's own issue body limit (65,536 characters). Currently deferred for consistency with existing providers.

## 1. Feature Overview

When `qode` CLI commands are invoked from an IDE extension (VS Code, Cursor, etc.), the process does not inherit the user's interactive shell environment. As a result, environment variables such as `GITHUB_TOKEN`, `JIRA_API_TOKEN`, `LINEAR_API_KEY`, and `AZURE_DEVOPS_PAT` are unavailable, causing commands like `qode ticket fetch` to fail with authentication errors.

This change adds transparent `.env` file loading at CLI startup. Before any command executes, `qode` looks for a `.env` file in the project root and loads it into the process environment, without overriding variables already present in the environment. This makes `qode` usable from IDEs without any additional shell configuration.

**Success criteria:**
- `qode ticket fetch <url>` succeeds when the required token is in `.env` but absent from the shell environment.
- No regressions when tokens are provided via shell exports or CI environment variables.
- Absence of `.env` is not an error.

```go
// Load reads a .env file from projectRoot and sets any variables that are
// not already present in the environment. If projectRoot is empty, the
// current working directory is used. A missing .env file is not an error.
func Load(projectRoot string) error
```

**Behaviour:**
- If `projectRoot` is empty, substitutes `os.Getwd()`.
- Constructs path: `filepath.Join(projectRoot, ".env")`.
- If the file does not exist (`os.IsNotExist`), returns `nil`.
- Calls `godotenv.Read(path)` to parse the file, then for each key, sets it via `os.Setenv` only if `os.Getenv(key) == ""`.
- If the file exists but cannot be parsed, returns the wrapped error (caller logs and continues).

**Why `godotenv.Read` + manual `os.Setenv` instead of `godotenv.Load`:**
`godotenv.Load` does not override existing vars, which is the desired behaviour. However, using `godotenv.Read` + manual loop gives explicit control and makes the non-override semantics visible in code.

## 1. Feature Overview

Currently, the command is not re-entrant: every invocation after the first generates an "Iteration 1" prompt instead of the correct next iteration number. // stdin/stdout/stderr are wired to the current terminal.
// Falls back to batch mode (c.Run) when stdin is not a terminal.
// The prompt is written to a temp file; a short read-instruction is

```

**Behaviour:**
- If `!isTTY()`: delegates to `c.Run(ctx, prompt, opts)`, discards the string return, returns the error.

## 8. Security Considerations

- **No shell injection risk.** `exec.CommandContext` is called directly (not via `sh -c`). User-controlled content (diffs, prompts) is written to a temp file and never interpolated into the argument list.
- **Temp file permissions.** `os.CreateTemp` creates files with mode `0600` by default on Unix. Prompt content is not sensitive (it is also written to `.qode/branches/` prompt files), but restrictive permissions are correct.
- **Temp file cleanup.** `defer os.Remove(f.Name())` ensures cleanup on all exit paths including panics.
- **`CLAUDECODE` env stripping.** Preserved in `RunInteractive` via existing `filterEnv(os.Environ(), "CLAUDECODE")`.
- **No authentication changes.** `claude` handles its own auth. `qode` does not handle credentials.
- **No network changes.** All network calls are made by `claude`, not by `qode`.

### [LOW-1] Success-Path Response Body Not Size-Limited

- **Severity:** Low
- **OWASP Category:** A05:2021 – Security Misconfiguration
- **Files:** `internal/ticket/github.go:112`, `internal/ticket/jira.go:82`, `internal/ticket/azuredevops.go:61`, `internal/ticket/linear.go:61`

// In each provider's decode call:
if err := json.NewDecoder(io.LimitReader(resp.Body, successBodyMaxBytes)).Decode(&result); err != nil {
```

### [LOW-2] `dangerTagRe` Does Not Handle Unclosed Dangerous Tags

- **Severity:** Low
- **OWASP Category:** A03:2021 – Injection
- **File:** `internal/ticket/azuredevops.go:93`
- **Vulnerability:** `dangerTagRe` (`(?is)<(script|...)>.*?</(script|...)>`) requires a matching closing tag. A ticket description containing `<script>malicious content` (no `</script>`) passes `dangerTagRe` unchanged; `htmlTagRe` then strips only the `<script>` opener, leaving `malicious content` verbatim in the output written to `ticket.md`. Since `ticket.md` is a Markdown file read by the AI and potentially rendered in a developer IDE, the risk of content injection is low but non-zero.
- **Remediation:** Add a second regex pass for unclosed dangerous tags:
```go
var (
    dangerTagRe      = regexp.MustCompile(`(?is)<(script|style|iframe|object|embed)[^>]*>.*?</(script|style|iframe|object|embed)>`)
    unclosedDangerRe = regexp.MustCompile(`(?is)<(script|style|iframe|object|embed)[^>]*>.*`)
    htmlTagRe        = regexp.MustCompile(`<[^>]+>`)
)

func stripHTML(s string) string {
    s = dangerTagRe.ReplaceAllString(s, "")
    s = unclosedDangerRe.ReplaceAllString(s, "")
    return htmlTagRe.ReplaceAllString(s, "")
}
```
